# CRuby: 普段のRubyの中身を覗く

普段 `ruby` コマンドで動かしてるRuby、正式名称は **CRuby**（C言語で書かれたRuby）。
「コードを書いて実行する」の裏側で何が起きてるか、ちょっと覗いてみよう。

---

## Step 0: セットアップ

```bash
cd ~/rubykaigi/cruby
rbenv local 3.4.5
ruby --version
# ruby 3.4.5 が出ればOK
```

---

## Step 1: Rubyのコードが動くまでの流れ

`puts "hello"` を実行すると、裏側ではこうなってる:

```
 あなたが書いたコード
       │
       ▼
   ① パース（解析）     ← 文字列をRubyが理解できる形に変換
       │
       ▼
   ② AST（抽象構文木）  ← コードの構造をツリーで表現
       │
       ▼
   ③ コンパイル         ← ツリーを命令の列に変換
       │
       ▼
   ④ バイトコード       ← VMが実行できる命令の列
       │
       ▼
   ⑤ YARV（VM）        ← 命令を1つずつ実行する
       │
       ▼
     結果！
```

「パース？AST？」ってなっても大丈夫。これから1つずつ見ていく。

---

## Step 2: バイトコードを見てみる（01_bytecode.rb）

Rubyのコードが「どんな命令に変換されるか」を見ることができる。

```bash
ruby 01_bytecode.rb
```

`RubyVM::InstructionSequence` を使うと、Rubyがコードをどういう命令に分解したか見える。
全部理解する必要はない。「あ、裏側ではこういう命令になるんだ」くらいでOK。

---

## Step 3: オブジェクトの正体を調べる（02_object_id.rb）

Rubyでは **全てがオブジェクト**。数字も文字列も配列も。
それぞれに固有のID（`object_id`）がある。

```bash
ruby 02_object_id.rb
```

面白いポイント:
- 小さい整数（0〜）は毎回同じobject_id → メモリを節約してる
- 文字列は毎回違うobject_id → 毎回新しく作られてる
- `freeze` すると変更不可になる

---

## Step 4: メソッド探索を追いかける（03_method_lookup.rb）

`"hello".upcase` と書くと、Rubyは `upcase` メソッドをどこから見つけてくるのか。

```bash
ruby 03_method_lookup.rb
```

Rubyがメソッドを探す順番:
1. そのオブジェクトの特異メソッド
2. そのクラスのメソッド
3. 親クラスのメソッド（継承チェーン）
4. `method_missing`

`ancestors` メソッドで「探す順番」が見える。

---

## Step 5: GC（ガベージコレクション）を観察する（04_gc.rb）

使わなくなったオブジェクトを自動的に片付けてくれる仕組み = GC。

```bash
ruby 04_gc.rb
```

大量にオブジェクトを作って、GCが動く様子を観察する。
`GC.stat` で「今どれくらいオブジェクトがあるか」が見える。

---

## ふりかえり

- **CRuby** = C言語で書かれたRubyの実装（普段使ってるやつ）
- コードは パース → AST → バイトコード → VM実行 の流れで動く
- `RubyVM::InstructionSequence` でバイトコードが見える
- 全てがオブジェクトで、`object_id` で確認できる
- メソッドは継承チェーンを辿って見つかる（`ancestors`で確認）
- GCが不要なオブジェクトを自動で片付けてくれる

この知識があると、この後のYJIT・ZJIT・Parserの話がもっと分かりやすくなる。
